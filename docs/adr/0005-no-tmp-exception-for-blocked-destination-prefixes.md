# ADR-0005: No `/tmp` exception for blocked move-to destination prefixes

- **Status**: Accepted
- **Date**: 2026-07-14
- **Plan**: `.claude/plans/2026-07-14-config-bug-batch.md`

## Context

Fixing #391 (`validate_destination`'s blocked-prefix check silently passing nonexistent paths
under a blocked system prefix) required deciding how to treat `/tmp`. On macOS, `/tmp` is a
symlink to `/private/tmp`, and `/private` is in `BLOCKED_DESTINATION_PREFIXES`
(`/usr /etc /System /Library /bin /sbin /var /private`) — so once #391's ancestor-walk
canonicalization correctly resolves symlinks in the existing-ancestor chain, any destination
under `/tmp` on macOS resolves under `/private` and is blocked. This directly affected the
README's own documented `move-to` example, `destination = "/tmp/omamori-quarantine/"`, which — it
turned out during investigation — was **already** non-functional on macOS at execution time
(`move_to_dir`'s runtime canonicalize already rejected it); #391's fix would also now reject it
at `config add`/load time, consistently.

Three options were considered for the *quarantine directory location* the README recommends and
the code accepts:

1. Add a code-level exception so `/tmp` (and its macOS `/private/tmp` resolution) is allowed
   through the blocked-prefix check, keeping the existing README example working as-is.
2. Leave the blocked-prefix list untouched, but change the recommended example (and generated
   config templates) to a location under the user's home directory instead.
3. Remove `/private` from the blocked list entirely.

## Decision

**Option 2**: no code-level exception. `/tmp`/`/private/tmp` remain blocked like every other path
under `/private`. The README, `config_template()` (generated by `omamori init`), and
`config.default.toml` now recommend `/Users/you/.omamori-quarantine/` (a home-directory path)
instead.

## Alternatives Considered

| Option | Rejected because |
|---|---|
| (1) `/tmp` allowlist exception | Opens a real attack surface: `move_to_dir`'s timestamped staging subdirectory is created with `create_dir_all` (non-exclusive) and has no symlink check of its own. Under a world-writable directory like `/tmp`, another local process could pre-plant a symlink at the predictable `<dest>/<unix-timestamp>` path, redirecting a subsequent quarantine `rename` to an attacker-controlled location — a threat class that does not exist for a destination under the user's own home directory. Independently, `/tmp` is conventionally wiped on reboot, which is a poor fit for a "moved here instead of deleted, in case you need it back" quarantine location — the value of `move-to` over `trash` is recoverability. Finally, carving out `/private/tmp` as an exception inside `BLOCKED_DESTINATION_PREFIXES`'s otherwise-uniform "system directory" semantics would need its own segment-aware special case, adding review surface to security-critical path-matching code for one path. |
| (3) Remove `/private` from the blocked list | `/etc`, `/var`, and other conventionally-important system directories are themselves symlinks to `/private/etc`, `/private/var` on macOS (firmlinks). Removing `/private` from the blocked list would reopen exactly the system-directory write protection those prefixes exist to provide. |

## Consequences

- Existing user configs that already reference a destination under `/tmp` continue to load (the
  post-load `validate_rules` path only *warns and disables that one rule*; it does not error out
  or block startup), so this is not a breaking change to existing installs — only to the
  documented example and to what `config add`/`init` recommend going forward.
- `move_to_dir`'s existing runtime fail-close behavior (reject if the resolved destination is
  under a blocked prefix) is unchanged and remains authoritative; the validate-time ancestor-walk
  check added for #391 is deliberately best-effort/non-authoritative and shares this same
  blocked-prefix list, so the two layers stay consistent by construction.
- Linux does not have this specific `/tmp` collision (`/tmp` is not a `/private` symlink there),
  so this decision only changes observed behavior on macOS — an intentional, accepted platform
  difference, not a new one introduced by this PR (Linux `/tmp` was already unaffected by
  `BLOCKED_DESTINATION_PREFIXES` before this fix).
- A residual, narrower version of the symlink-planting risk described above still exists if a
  user explicitly configures a `move-to` destination under *any* other world-writable, shared
  directory (not just `/tmp`) — hardening `move_to_dir`'s own timestamped-subdirectory creation
  to be exclusive (`create_dir` instead of `create_dir_all`, with an explicit symlink check) is
  tracked as separate follow-up work, out of scope for this fix
  ([#410](https://github.com/yottayoshida/omamori/issues/410)).
